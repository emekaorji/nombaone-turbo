# Docs — Plan 02 · Information architecture, navigation, voice & vocabulary
> Lock the Diátaxis IA, the manifest-driven navigation system, stable URLs, the OpenAPI-derived glossary, the house style guide, and a CI-failing Vale vocabulary pack — so the docs have one skeleton and one voice before any content lands. Prerequisites: Phase 00 (contract). Serves tenets: 5 (the API is the product / one glossary), 6 (docs are the demo / agent-readable), 8 (honest hard parts have a home in the IA), 10 (one platform, one language, one voice).

## Goal

Make `content/manifest.ts` the single ordered source of truth for the entire docs surface, covering every §5 Diátaxis section with intentional ordering, Diátaxis-mode tags, and badges; wire the chrome (sidebar, breadcrumbs, right-rail TOC, prev/next pager) to read from it via a flattened tree; guarantee stable canonical URLs that never 404 (with a redirects map); generate a one-noun-per-concept glossary from the OpenAPI spec where every domain noun links to its schema; codify the house style in `STYLE.md`; and ship a Vale rule-pack that fails CI on any violation of the locked vocabulary (`:id` not `:ref`, `organizations` not `tenants`, `/webhooks` + nested deliveries, integer kobo, no marketing fluff in reference, sentence-case headings, second person / active voice / present tense).

This phase writes **structure and rules**, not page bodies. Every section defined here is a placeholder route that later phases fill (numbers cross-referenced, never duplicated). The success condition is: an implementer can add any page in any later phase by appending one manifest entry, and the nav, breadcrumbs, TOC, pager, glossary linkage, and prose lint all "just work" and stay honest.

### B · Manifest-driven navigation behavior

- [ ] **Wire the sidebar to the manifest** in `src/components/chrome/sidebar-nav.tsx` (consume, do not restyle — Phase 01 owns visuals). Render `MANIFEST` as collapsible sections; each section collapses/expands with its open state persisted to `localStorage` keyed by section slug; the active item is derived from the current route via `findItem`, and its ancestor sections auto-expand on load. **Acceptance:** active item has `aria-current="page"`; keyboard: arrow/enter navigate and toggle, focus ring ≥2px at 3:1 (§4 a11y); collapse state survives reload; no layout shift on expand.
- [ ] **Implement API-section density mode** in the sidebar for `api-reference`: when inside `api/*`, render operations as compact rows with a leading **method chip** (color-coded but a11y-labeled, not color-only — GET/POST/PUT/PATCH/DELETE each get a text label for screen readers), grouped under their module, with the module collapsible. **Acceptance:** the dense API list is scannable at ~80 operations without the sidebar becoming unusable; method chips meet 3:1 contrast on the surface and carry `aria-label` (e.g. "POST"); the current operation's module stays expanded.
- [ ] **Wire breadcrumbs to `breadcrumbTrailFor`** in `src/components/chrome/breadcrumbs.tsx`. Render Section › (Module ›) Page, each crumb a real link. **Acceptance:** crumbs match the manifest trail exactly; the last crumb is `aria-current`; links are descriptive (§4 "descriptive links"); no crumb 404s.
- [ ] **Wire the right-rail TOC** in `src/components/chrome/toc.tsx` to headings extracted at build time by the MDX pipeline (h2/h3), with a scroll-spy active-heading highlight. **Acceptance:** TOC reflects only h2/h3 of the current page; active heading updates on scroll via `IntersectionObserver`; TOC is keyboard-navigable and hidden (not just visually) from the a11y tree when empty.
- [ ] **Wire the prev/next pager** in `src/components/chrome/pager.tsx` to `FLAT_NAV[i].prev/next`. Show the neighboring page titles and section labels; wrap across section boundaries. **Acceptance:** pager order equals the flattened manifest order end-to-end; first page has no prev, last has no next; links descriptive; verified by a snapshot test asserting the full ordered slug sequence.
- [ ] **Add a manifest-driven "section landing" convention.** Each `ManifestSection` root slug (e.g. `/guides`) renders an auto-generated index of its items (title + summary cards) so no section root 404s even before its overview page is authored. **Acceptance:** visiting any section root slug returns a real page listing that section's items; when an explicit `overview` item exists it takes precedence.

- [ ] **Create the redirects map** `apps/docs/content/redirects.ts` exporting `REDIRECTS: { from: string; to: string; permanent: boolean }[]` and wire it into the Next config's `redirects()` (or the catch-all route's early redirect) so any renamed/removed slug 301s to its successor. Seed it with the legacy `example` reference page → the new module route, and any getting-started slug changes from Phase 01. **Acceptance:** a request to a retired slug 301s (not 404s); a CI test (`scripts/check-links.ts`, task E) asserts no `to` target is itself a 404 and no `from` collides with a live manifest slug.
- [ ] **Add a build-time "no orphan, no 404" link checker** `apps/docs/scripts/check-links.ts` that (1) collects every manifest slug + every heading anchor, (2) crawls all `content/**/*.mdx` internal links + every `docUrl`-shaped reference, and (3) fails the build if any internal link, redirect target, or manifest slug points nowhere. **Acceptance:** wired as a peer of the docs `type-check` turbo task; red on any broken internal link; explicitly asserts `/errors` resolves (guards the API's promise).

### D · Glossary generated from the OpenAPI spec

- [ ] **Author the glossary generator** `apps/docs/scripts/gen-glossary.ts`. It imports `buildOpenApiDocument` from `@nombaone/api` **at build time (no network)** when available; walks `components.schemas`; and emits `content/_generated/glossary.json` — one entry per canonical domain noun: `{ term, definition, schemaRef, schemaUrl, conceptUrl, aliases: string[] }`. **One noun per concept** (§10 anti-synonym rule): the generator asserts no two schemas map to two names for the same concept and fails if it finds the Paystack-style tangle. **Fallback:** if `@nombaone/api` is not importable yet (this branch), read a checked-in seed `content/glossary.seed.ts` so the build still produces a glossary. **Acceptance:** produces entries for at least: organization, plan, price, cycle, invoice, subscription, customer, payment method, mandate, settlement, sub-account, coupon, credit, dunning, refund, payout, webhook, event, delivery, idempotency key, reference, escrow, kobo. Each `schemaUrl` deep-links to that schema in the API reference (Phase 08 anchor convention `api/<module>#schema-<Name>`); each `conceptUrl` links to its Concepts page. Regenerating twice is deterministic (stable sort).
- [x] **Author the glossary seed** `apps/docs/content/glossary.seed.ts` with the canonical noun list above, each with a one-sentence sentence-case definition in house voice and its target `schemaRef` name (e.g. `Organization`, `Subscription`, `Invoice`, `Mandate`, `Settlement`). **Acceptance:** the seed is the single source for the anti-synonym Vale rule (task F reads the same list); definitions pass the Vale reference-voice rules (no marketing adjectives); `kobo` entry states "the integer minor unit; ₦1 = 100 kobo; the API is always integer kobo".
- [ ] **Add the `<GlossaryTerm>` MDX affordance + auto-link pass.** Create `src/components/mdx/glossary-term.tsx` (`'use client'` leaf) that renders a term with a hover/press definition popover sourced from `glossary.json`, and a rehype transformer `src/lib/rehype-glossary.ts` that, on first occurrence per page, links a canonical noun to `/concepts#<term>` and its schema. **Acceptance:** every domain noun in prose links to its glossary/schema on first use; the popover is keyboard-openable and dismissible (Esc), meets AA contrast on dark; auto-linking never double-wraps inside code or existing links.
- [x] **Render the glossary page** `content/reference/glossary.mdx` (or a generated route) listing all terms alphabetically, each linking to schema + concept, with aliases shown as "also seen as … (we use `term`)". **Acceptance:** the page is in the manifest under `api-reference` or `concepts` (pick `api-reference` for information-orientation), lists every generated term, and every schema link resolves (checked by task C's link checker).

## Acceptance criteria (how we know the phase is done — testable)

- **One source of truth:** every route the docs will ever have is reachable from `content/manifest.ts`; the sidebar, breadcrumbs, TOC, and pager render entirely from it; adding a page = one manifest entry with no other wiring.
- **Diátaxis is not mixed:** `manifest.test.ts` proves no section holds items of a foreign `mode`; the four modes map to the ten sections exactly per §5.
- **Nothing 404s:** `check-links.ts` is green — no internal link, redirect target, section root, or manifest slug points nowhere; `/errors` resolves (guarding the live API's `docUrl` promise); a request to a retired slug 301s.
- **Pager walks the whole book:** a snapshot test asserts `FLAT_NAV`'s ordered slug sequence and reciprocal prev/next across section boundaries.
- **One glossary, one noun per concept:** `gen-glossary.ts` produces an entry for every canonical noun; the anti-synonym assertion passes (no concept has two names); every noun links to its schema and its concept; `<GlossaryTerm>` auto-links first occurrences and is keyboard-accessible at AA contrast.
- **One voice enforced by CI:** `turbo run lint:prose` fails the build on `:ref`, `tenant`, `/webhook` (singular), a naira amount in a body, a Title-Case heading, or a marketing adjective in reference; `vale-selftest.sh` proves the pack works; `STYLE.md` is itself lint-clean (dogfooded).
- **A11y holds on dark:** sidebar/breadcrumbs/TOC/pager are fully keyboard-operable, active states use `aria-current`, method chips carry text labels (not color-only), focus rings ≥2px at ≥3:1 (§4).
- **Agent-readable structure:** the manifest's typed `mode`/`summary`/`badges` and the generated glossary are consumable by Phase 09's `.md` mirror + `llms.txt` without rework (frontmatter/typing aligns with `apps/website/content/hard-parts/frontmatter.ts`).

## Antipattern watch (the §10 rules most relevant here)

- ❌ **Blending Diátaxis modes on a page** — enforced by the section `mode` invariant + `manifest.test.ts`. A tutorial must not carry reference tables; Hard Parts stay in `concepts`/Explanation.
- ❌ **Broken/redirecting canonical URLs / 404s** (Linear `/docs`→`/developers`, WorkOS `/docs` 404) — the URL policy freezes shipped slugs, `redirects.ts` handles renames, `check-links.ts` blocks orphans, and `/errors` is explicitly guarded.
- ❌ **Reference proliferation of near-synonym IDs** (`tx_ref`/`flw_ref`/`trxref`) — the glossary's one-noun-per-concept assertion + the `Ref.yml`/Vocab rules make a second name for a concept a build failure.
- ❌ **Marketing fluff / Title Case / passive / future tense in reference** — `NoFluff.yml` + `SentenceCase.yml` + the Google/Microsoft base pack fail CI on these.
- ❌ **Naming money fields without pinning the unit** — `Money.yml` fails on naira-in-body and heuristically flags the 100× trap.
- ❌ **Dark mode without contrast math** — nav a11y bars (focus ≥2px @3:1, `aria-current`, non-color-only method chips) are acceptance-gated.
- ❌ **Product-bundling-first IA that buries the quickstart** (Prisma) — `get-started`/`quickstart` is section one, item two; the IA is organized by the developer's goal, not by SKUs.

## Manifesto ties

- **Tenet 10 — one platform, one language, one voice:** the Vale pack + `STYLE.md` + the one-noun glossary make "one vocabulary" machine-enforced across the docs (and the shared basis for api/console/checkout); a term cannot mean two things without failing CI.
- **Tenet 5 — the API is the product:** the glossary is generated from the OpenAPI schemas and every noun links to its schema; the API-reference tree is authored as the honesty-gate's reconciliation target.
- **Tenet 6 — docs are the demo / if it isn't in the docs it doesn't exist:** the manifest reserves a real, non-404 home for all ~80 operations, 40+ events, and ~130 errors, and the typed structure is agent-consumable for Phase 09.
- **Tenet 8 — honest about the hard parts:** the IA gives Hard Parts a first-class, badge-marked home in Explanation, wired to the 17 real essays by their exact slugs.
- **Tenet 7 — merchants deserve the same care:** the `merchants` no-code track is named by what merchants control (plan, payment link, dunning message, settlement), under the one design system and voice.
