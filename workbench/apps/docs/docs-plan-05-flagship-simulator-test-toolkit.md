# Docs — Plan 05 · The flagship simulator + test toolkit

> Build the one thing no payments doc has: an in-page explorable where a reader **watches** a Nigerian subscription bill, fail on a thin balance, dun, and recover — driven by our REAL `/v1/test/*` instruments, streaming REAL signed webhooks into an inline inspector, with no login. Prerequisites: 03 (proxy → `/v1/test/*` + SSE, ephemeral key), 04 (`/errors#CODE` + auto-link). Serves tenets: 4 (curiosity → watch it happen), 8 (honest about the hard parts), 9 (errors are a feature), 1 (the money is never wrong), 6 (docs are the demo).

## Goal

Ship the single most differentiated asset in the docs: the `<LifecycleSimulator>` island and the pages that host it. A reader — anonymous, no signup — clicks through a stepped runner that, through the Phase-03 proxy/SSE against the **real test sandbox**, mints a deterministic test payment method, creates a subscription, advances the billing clock, forces a thin-balance decline or an OTP wall, watches dunning fire, and lands on `invoice.payment_recovered` in emerald. Every step streams its genuine request/response (with `X-Request-Id`) into an expandable inline **event inspector**, and animates the existing `<LifecycleStateMachine>` and `<MoneyFlow>` islands from LIVE data — not authored props. A "Place Your Bets" gate (Nicky Case) makes the reader predict recovery before advancing. Alongside it: the **Test toolkit** reference page documenting the three instruments + each behavior token with a one-click try-it, and the **Nigerian test-method registry** (a searchable token → outcome → error code → branch-to-write table). This phase also **de-examples** the two data islands: their presets become the real subscription/invoice/dunning lifecycles, never the throwaway `example`.

Nothing here fakes anything. If a call can't be run for real (test base absent, ephemeral key not minted), the UI says so plainly and disables the run — it never fabricates a response.

## Prerequisites (what must exist first)

- **Phase 03 — the runnable spine.** This phase is a pure consumer of it. Specifically it needs:
  - The playground proxy (`apps/docs/src/app/api/playground/route.ts`) allowlist extended to the `/v1/test/*` paths (`/test/payment-methods`, `/test/subscriptions/:id/advance-cycle`, `/test/webhooks/simulate`) **and** the read/create paths the simulator drives (`/subscriptions`, `/customers`, `/plans`, `/prices`, `/payment-methods`, `/invoices`), all schema-driven from the in-repo OpenAPI, **test-base-only guarded** (the proxy must refuse to forward `/test/*` to any base that is not the test sandbox).
  - The **SSE proxy variant** (`apps/docs/src/app/api/playground/stream/route.ts` or equivalent) so a multi-step run streams request/response frames as they complete.
  - The **ephemeral sandbox key + throwaway org** route (tenet 14) so the simulator runs for an anonymous reader; and a **per-session signed-webhook capture sink** the ephemeral org's webhook endpoint points at, whose deliveries stream to the browser (see the "webhook capture" deliverable — Phase 05 builds the sink UI/stream but relies on Phase 03 minting the org + pointing its webhook at our sink URL).
  - `LIVE_KEY_REJECTED` / `nbo_test_` enforcement and `X-Request-Id` pass-through already present in the proxy (they are — confirmed in `route.ts`).
- **Phase 04 — the error reference.** `/errors#CODE` pages exist for every `PUBLIC_ERROR_CODES` entry, and the rehype auto-linker turns error tokens in fenced blocks into deep links. The registry's "error code" column and the inspector's hint links point here; if 04 is not done, these are 404s and this phase is not done.
- **Phase 01/02 — chrome, tokens, manifest.** The emerald accent fix (`--accent` `#0bdfa3`/`#00a473`) landed; `content/manifest.ts` shape + section helpers exist so we can register the Test toolkit section.
- **In-repo facts this phase is built against (read, never modified from docs):**
  - Instruments: `apps/api/src/apps/main/modules/test/routes.ts` + controllers (`create-test-payment-method.ts`, `advance-cycle.ts`, `simulate-webhook.ts`).
  - Behavior tokens + bodies: `packages/core-contracts/src/validations/test.ts` — `testMethodBehaviors = [success, decline_insufficient_funds, decline_expired_card, decline_do_not_honor, requires_otp]`, `kind ∈ {card, mandate}`; `createTestPaymentMethodBody { customerId, behavior, kind }`; `simulateWebhookBody { type, payload? }`.
  - Deterministic outcomes: `packages/sara/src/rails/test-sim.ts` — sentinels `test_success | test_decline_insufficient_funds | test_decline_expired_card | test_decline_do_not_honor | test_requires_otp` → `succeeded | failed(failureReason) | requires_action(otp_3ds)`.
  - Event catalog: `packages/core-contracts/src/types/webhook-events.ts` — `WEBHOOK_EVENT_CATALOG` (the exact event strings + `when` + payload keys, incl. `invoice.action_required` carrying `checkoutLink`), `WEBHOOK_DELIVERY_GUARANTEE = 'at-least-once'`, header `x-nombaone-delivery-guarantee`.
  - Islands to reuse/extend: `apps/docs/src/components/mdx/{lifecycle-state-machine.tsx, money-flow.tsx, api-explorer.tsx}`.

### B. `<LifecycleSimulator>` — the flagship island

- [ ] **Create `apps/docs/src/components/mdx/lifecycle-simulator.tsx`** (`'use client'` leaf island). It is a **stepped runner** with a declarative `steps` prop (a preset scenario id, e.g. `scenario="thin-balance-recovery" | "otp-wall" | "clean-renewal" | "churn"`) resolving to an ordered step list. Each step declares: a human title, the real call it makes (method + `/v1/test/*` or resource path + body), the expected resulting event(s) from `WEBHOOK_EVENT_CATALOG`, and the island animation it drives (`stateMachine.activeState`, `moneyFlow.revealTo`). **Acceptance:** the component renders the step list with a clear current-step indicator and a single primary action per step; it holds no business logic that duplicates the server — it only orchestrates calls and renders what comes back.
- [ ] **Wire the runner to the Phase-03 proxy/SSE.** Each step POSTs through `/api/playground` (single calls) or subscribes to the SSE stream route (multi-frame steps). The reader's key comes from the shared `localStorage` key (`nombaone-docs:test-api-key`, same key `<ApiExplorer>` uses) **or** the Phase-03 ephemeral key minted on first run (the "no login" path). **Acceptance:** with no key present, a "Run without signing up" affordance mints an ephemeral test key via the Phase-03 route and proceeds; a pasted `nbo_live_` key is refused in-UI with the same copy the proxy uses (defence in depth); every network call is visible in the reader's own devtools hitting the same-origin proxy (never the sandbox directly).
- [ ] **Implement the canonical `thin-balance-recovery` scenario** as the default: (1) create/get a demo customer → (2) `POST /v1/test/payment-methods { behavior: "decline_insufficient_funds", kind: "card" }` → (3) create a subscription on a real plan/price → (4) `POST /v1/test/subscriptions/:id/advance-cycle` (the invoice bills and the charge fails on the thin balance) → (5) show dunning (`invoice.payment_failed`, `subscription` → `past_due`) → (6) swap to a recovering method (`behavior: "success"`) or `POST /v1/test/webhooks/simulate { type: "invoice.payment_recovered" }` per the built engine's real recovery path → (7) land on `invoice.payment_recovered` + `subscription.activated`, the state machine lit emerald. **Acceptance:** every response shown is the genuine upstream envelope; the terminal state is reached only when the real `invoice.payment_recovered` event is observed, not on a timer.
- [ ] **Implement the `otp-wall` scenario** using `behavior: "requires_otp"`: the charge returns `requires_action (otp_3ds)`, the invoice flips to `action_required`, and the inspector surfaces the `invoice.action_required` event **with its `checkoutLink`** — teaching our live-confirmed OTP/3DS recharge wall and the dunning-to-checkout fallback. **Acceptance:** the `checkoutLink` from the real event is rendered as the branch a developer takes; the page is explicit that headless completion is not possible (cross-link the Phase-07 OTP wall page by number, do not duplicate its prose).
- [ ] **Implement a `churn` scenario** (dunning exhausted → `subscription.churned`) so the unhappy terminal path gets equal billing to recovery (§4: unhappy paths get disproportionate care). **Acceptance:** the terminal `churned` state is reached from the real exhausted-dunning path and rendered in `danger` tone, not hidden.
- [ ] **Drive the islands from live data.** As each step's real response/events arrive, set `<LifecycleStateMachine activeState=…>` and `<MoneyFlow revealTo=… legs={liveLegs}>` from the actual payload (invoice status, ledger legs). **Acceptance:** the two islands reflect the sandbox's real state at each step; if the sandbox returns an unexpected shape, the island shows the raw payload rather than a fabricated animation.
- [ ] **Test-only + failure guardrails.** If `NEXT_PUBLIC_INFRA_API_BASE` is not a test base, or the ephemeral-key/proxy call fails, the runner renders a plain, honest disabled state ("This runs against the test sandbox — unavailable here") and never simulates a response. **Acceptance:** flipping the base to a non-test value disables Run; a proxy 5xx surfaces the real error envelope + its `/errors#CODE` link, never a mock.
- [ ] **a11y + reduced motion.** The runner is keyboard-operable (each step's action is a real `<button>`; the step list is an ordered list with an `aria-current` step); a polite `aria-live` region announces each step's outcome; under `prefers-reduced-motion` the island animations resolve to their end state instantly (reuse the existing island gates). Focus rings use the emerald `--ring` at ≥2px, 3:1. **Acceptance:** full run completable by keyboard only; SR announces "Step 4 of 7: charge failed — insufficient funds"; reduced-motion shows no timed sequencing.

- The **event inspector** shows only real data: genuine request/response envelopes with real `X-Request-Id`, and real signed webhook deliveries carrying `x-nombaone-delivery-guarantee: at-least-once`; the sink rejects a tampered body.
- Switching scenarios exercises `otp-wall` (surfaces the real `invoice.action_required.checkoutLink`) and `churn` (reaches `subscription.churned` in danger tone) — the unhappy paths are as prominent as recovery.
- The **Test toolkit** page documents all three instruments with fields matching the OpenAPI (honesty gate green); each has a working one-click `<ApiExplorer>`; the five behavior tokens each demonstrate their deterministic outcome.
- The **registry** filters client-side; every built token matches a `test-sim.ts` sentinel and links to a live `/errors#CODE`; proposed tokens are clearly labelled and are **not** runnable.
- The two data islands render the **real** subscription/mandate lifecycles and a real kobo ledger flow — **zero** references to the `example` resource on any Phase-05 page; no `purple-` class remains; both pass the a11y contrast + focus checks (§4).
- Flipping `NEXT_PUBLIC_INFRA_API_BASE` to a non-test value (or a proxy failure) disables Run and shows an honest message + the real error — never a mocked response.
- The full simulator run is completable by **keyboard only**, with SR announcements per step and reduced-motion resolving animations to their end state.
- The three new slugs never 404 and are wired into the sidebar + pager; the honesty gate + type-check + lint are green.

## Antipattern watch (the §10 rules most relevant here)

- ❌ **Faked simulations / mocks.** Every step hits the real sandbox via the proxy + `/v1/test/*`. No `setTimeout` theatre standing in for a response; timers only pace *revealing* data that already arrived. A payments audience opens devtools.
- ❌ **Breaking the `/v1/test/*` test-env guard.** The runner must never forward test instruments to a non-test base; guard in-UI and rely on the proxy's test-base-only guard. Advance-cycle bills a real cycle — it is test-only by construction.
- ❌ **Live keys / live money.** `nbo_live_` is refused in-UI (defence in depth over the proxy's server-side rejection); the ephemeral key is test-scoped.
- ❌ **Calling the sandbox directly from the browser.** All traffic goes through the same-origin proxy/SSE — never a client-direct call.
- ❌ **Presets frozen on the throwaway `example`.** The islands are de-examples to real subscription/invoice/dunning data; the `example` preset survives only for the deletable scaffold and is unused here.
- ❌ **Happy-path-only.** `otp-wall` and `churn` scenarios ship alongside recovery; the registry documents declines/OTP/dead-mandate; the state machine renders terminal `churned`/`action_required`.
- ❌ **Error hints pointing at a 404.** Every hint/error-code in the inspector + registry deep-links to a live Phase-04 `/errors#CODE` page.
- ❌ **Faking the "proposed" tokens.** Un-built tokens are labelled proposed and are **not** wired to a live try-it (they would 400 upstream — that would be a lie).
- ❌ **Reference that isn't gate-checked.** The instruments' fields are validated against the in-repo OpenAPI; drift turns the build red.
- ❌ **Dark mode without contrast math / sandbox overload.** Emerald surfaces + focus rings meet AA on `#040404`; the Place-Your-Bets gate keeps the runner to a guided 3–5 step path before free exploration.

## Manifesto ties

- **Tenet 4 (curiosity → first success in minutes).** The reader watches the core loop happen — and verifies it in their own devtools — before writing a line, with no signup.
